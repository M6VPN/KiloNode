# M2.8 External Modem Roadmap

M2.8 records the modem support path for hobbyist releases. It does not add a
modem adapter implementation by itself.

## Goal

KiloNode should support practical external modem workflows without embedding
modem source code or mixing modem protocol logic into AX.25 state handling.

The near-term target is clear documentation for:

- Dire Wolf TCP KISS receive-only use.
- VARA FM and VARA HF as future external modem adapters.
- Mercury OFDM as a future external TCP modem adapter.
- ARDOP as a future external TCP host adapter.

## Support Tiers

| Tier | Scope | Status |
| ---- | ----- | ------ |
| 1 | KISS receive sources, including Dire Wolf and KiloTNC | current receive path |
| 2 | External TCP modem adapters for VARA, Mercury, and ARDOP | planned |
| 3 | Lab-only TX and connected sessions through external modems | future gated work |

## Boundaries

- External modems run as separate programs.
- KiloNode talks to documented process or TCP interfaces only.
- No modem source is vendored, copied, linked, or embedded.
- No TX, PTT, or live CONNECT support is enabled by this milestone.
- AX.25 connected-mode state remains inside KiloNode.
- FX.25 remains a separate future layer.

## Modem Notes

Dire Wolf is the first practical hobbyist receive path because it can expose
TCP KISS and KiloNode already handles TCP KISS receive workflows.

VARA FM and VARA HF are future adapter targets. They should be treated as
external applications with a small KiloNode adapter around their documented
interfaces.

Mercury is a Rhizomatica HF OFDM modem project with compatible TCP interfaces.
KiloNode should treat Mercury as an external modem boundary and should not copy
or reuse Mercury implementation code.

ARDOP has a TCP host interface used by external client programs. KiloNode
should start with status/config documentation and later add a small adapter if
the interface is needed for receive or lab-only workflows.

## Next Implementation Work

The next code milestone after this roadmap should be a small external modem
configuration and status scaffold:

- modem type
- host and port
- enabled flag
- read-only status output
- no transmit commands
- no automatic process launch
- no live CONNECT exposure
